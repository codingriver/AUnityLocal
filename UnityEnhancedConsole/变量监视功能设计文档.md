# 变量监视功能设计文档

> UnityEnhancedConsole — Variable Watch Feature Design
> 状态：设计阶段 | 版本：v1.0

---

## 1. 概述

### 1.1 背景

变量监视（Variable Watch）允许开发者在运行时跟踪变量值的变化，无需反复调用 `Debug.Log`，避免日志污染。此功能是 Editor Console Pro 等商业工具的核心卖点之一。

### 1.2 目标

- 提供轻量级的变量监视 API，开发者可在代码中注册要监视的变量
- 在 EnhancedConsole 窗口中提供独立的监视面板，实时显示变量当前值
- 支持变化历史记录，可查看变量值的时间线
- 零分配（Zero Allocation）设计，不影响游戏运行时性能
- 与现有 Tag / 搜索 / 过滤系统集成

### 1.3 竞品参考

| 特性 | Editor Console Pro | LiveWatch | 本方案（设计目标） |
|------|-------------------|-----------|----------------|
| API 风格 | `Debug.Watch(name, value)` | `Watch.GetOrAdd<T>(name)` + Push | 静态方法 + Lambda 自动更新 |
| 值类型支持 | 基本类型 | 任意类型 + 集合 | 任意类型（ToString + 自定义格式化） |
| 变化历史 | ❌ | ✅ 完整历史 | ✅ 环形缓冲区（可配置深度） |
| 图表模式 | ❌ | ✅ | ✅（数值类型） |
| 条件高亮 | ❌ | ✅ | ✅ 阈值规则 |
| 分组 | ❌ | ❌ | ✅ 命名空间分组 |
| 持久化 | ❌ | ✅ 二进制 | ✅ JSON（复用现有日志文件机制） |
| 性能开销 | 低 | 极低（代码生成） | 低（反射缓存 + 条件编译） |

## 2. 用户 API 设计

### 2.1 核心 API

```csharp
namespace UnityEnhancedConsole
{
    /// <summary>
    /// 变量监视入口，所有 API 均为线程安全。
    /// 编辑器外构建中通过条件编译移除，零开销。
    /// </summary>
    public static class Watch
    {
        // ── 手动推送（最简单，类似 Debug.Log 但不污染日志） ──

        /// <summary>推送一个变量的当前值</summary>
        /// <param name="name">监视名称，支持 "/" 分隔的分组路径，如 "Player/Health"</param>
        /// <param name="value">当前值</param>
        [System.Diagnostics.Conditional("UNITY_EDITOR")]
        public static void Set(string name, object value);

        /// <summary>推送带格式化的值</summary>
        [System.Diagnostics.Conditional("UNITY_EDITOR")]
        public static void Set(string name, object value, string format);

        /// <summary>推送数值类型（避免装箱）</summary>
        [System.Diagnostics.Conditional("UNITY_EDITOR")]
        public static void Set(string name, float value);

        [System.Diagnostics.Conditional("UNITY_EDITOR")]
        public static void Set(string name, int value);

        [System.Diagnostics.Conditional("UNITY_EDITOR")]
        public static void Set(string name, bool value);

        [System.Diagnostics.Conditional("UNITY_EDITOR")]
        public static void Set(string name, Vector3 value);


        // ── 自动监视（注册 Lambda，每帧自动采集） ──

        /// <summary>注册自动监视，每帧调用 getter 获取最新值</summary>
        /// <param name="name">监视名称</param>
        /// <param name="getter">值获取函数</param>
        /// <param name="owner">拥有者对象（用于自动清理，可选）</param>
        [System.Diagnostics.Conditional("UNITY_EDITOR")]
        public static void Auto(string name, Func<object> getter, UnityEngine.Object owner = null);

        /// <summary>注册数值自动监视（避免装箱）</summary>
        [System.Diagnostics.Conditional("UNITY_EDITOR")]
        public static void Auto(string name, Func<float> getter, UnityEngine.Object owner = null);


        // ── 管理 ──

        /// <summary>移除指定监视</summary>
        [System.Diagnostics.Conditional("UNITY_EDITOR")]
        public static void Remove(string name);

        /// <summary>移除某个 owner 下的所有监视</summary>
        [System.Diagnostics.Conditional("UNITY_EDITOR")]
        public static void RemoveAll(UnityEngine.Object owner);

        /// <summary>清除所有监视</summary>
        [System.Diagnostics.Conditional("UNITY_EDITOR")]
        public static void Clear();

        /// <summary>暂停/恢复指定监视的采集</summary>
        [System.Diagnostics.Conditional("UNITY_EDITOR")]
        public static void SetPaused(string name, bool paused);
    }
}
```

### 2.2 使用示例

```csharp
// ── 场景 1：手动推送（最常用） ──
void Update()
{
    Watch.Set("Player/Health", _health);
    Watch.Set("Player/Position", transform.position);
    Watch.Set("FPS", 1f / Time.deltaTime);
}

// ── 场景 2：自动监视（注册一次，自动采集） ──
void OnEnable()
{
    Watch.Auto("Player/Health", () => _health, this);
    Watch.Auto("Player/Speed", () => _rigidbody.velocity.magnitude, this);
}

void OnDisable()
{
    Watch.RemoveAll(this);  // 自动清理
}

// ── 场景 3：分组路径 ──
Watch.Set("Network/Ping", pingMs);
Watch.Set("Network/PacketLoss", lossPercent);
Watch.Set("Physics/RaycastCount", raycastCount);
```

### 2.3 条件编译策略

```csharp
// 所有公开方法标记 [Conditional("UNITY_EDITOR")]
// 在非编辑器构建中，调用点被编译器完全移除
// 无运行时开销、无字符串分配、无装箱

// 对于 Auto() 的 Lambda 参数：
// [Conditional] 会移除整个调用（包括参数求值），所以 Lambda 不会被创建
```

## 3. 数据模型

### 3.1 核心数据结构

```csharp
/// <summary>单个监视条目</summary>
public class WatchEntry
{
    public string Name;              // 监视名称（含分组路径）
    public string Group;             // 分组名（从 Name 解析，"/" 之前的部分）
    public string DisplayName;       // 显示名（"/" 之后的部分）

    public object CurrentValue;      // 当前值
    public string FormattedValue;    // 格式化后的字符串表示
    public string Format;            // 自定义格式字符串（可选）

    public WatchValueType ValueType; // 值的类型分类
    public bool IsPaused;            // 是否暂停采集

    // 自动监视相关
    public Func<object> Getter;      // 自动采集函数（null = 手动推送模式）
    public WeakReference<UnityEngine.Object> Owner; // 弱引用拥有者

    // 历史记录（环形缓冲区）
    public WatchHistoryEntry[] History;
    public int HistoryHead;          // 环形缓冲区头指针
    public int HistoryCount;         // 当前历史条目数

    // 变化检测
    public string PreviousFormattedValue;
    public double LastChangeTime;    // EditorApplication.timeSinceStartup
    public int ChangeCount;          // 总变化次数
}

/// <summary>历史记录条目</summary>
public struct WatchHistoryEntry
{
    public double Timestamp;         // EditorApplication.timeSinceStartup
    public int FrameCount;
    public string FormattedValue;
    public float NumericValue;       // 仅数值类型有效（用于图表绘制）
    public bool HasNumericValue;
}

/// <summary>值类型分类（决定显示和图表支持）</summary>
public enum WatchValueType
{
    String,     // 默认，ToString() 显示
    Integer,    // int/long — 支持图表
    Float,      // float/double — 支持图表
    Boolean,    // bool — 支持状态指示器
    Vector,     // Vector2/3/4 — 支持分量图表
    Color,      // Color — 支持色块预览
    Object,     // 复杂对象 — ToString() 显示
}
```

## 5. 历史记录与图表

### 5.1 历史采集

- 使用环形缓冲区（Ring Buffer）存储历史，避免内存膨胀
- 默认深度 300 条（约 5 秒 @ 60fps），可在设置中调整到最大 10000
- 仅在值发生变化时记录新条目（变化检测基于 `FormattedValue` 字符串比较）
- 每条记录包含：时间戳、帧号、格式化值、数值（用于图表）

### 5.2 图表渲染

- 仅对数值类型（int, float, Vector 分量）启用图表
- 使用 UI Toolkit 的 `generateVisualContent` 自绘制（MeshGenerationContext）
- X 轴：时间（最近 N 秒），Y 轴：值范围（自动缩放）
- Vector 类型：X/Y/Z 分量分别用 红/绿/蓝 线条绘制
- 支持鼠标悬停显示精确值（Tooltip）
- 图表区域可拖拽缩放时间范围

### 5.3 历史列表模式

```
┌──────────┬────────┬──────────────┐
│ 时间戳     │ 帧号    │ 值            │
├──────────┼────────┼──────────────┤
│ 14:23:05 │ #12451 │ 85.3         │
│ 14:23:04 │ #12389 │ 82.1         │
│ 14:23:04 │ #12370 │ 80.0         │
│ ...      │        │              │
└──────────┴────────┴──────────────┘
```

## 6. 与现有系统集成

### 6.1 与日志系统的关系

- 监视面板作为独立 Tab，不与日志列表混合
- 监视数据**不**写入日志流（不产生 LogEntry），避免污染
- 但可以选择性地将监视快照导出为日志条目（用户主动操作）

### 6.2 与 Tag 系统集成

- Watch 条目的分组路径可作为虚拟 Tag 出现在 Tag 栏中
- 例如 `Watch.Set("Player/Health", v)` 产生 "Player" 分组，可在 Tag 栏中过滤

### 6.3 与导出系统集成

- 监视历史可通过导出菜单导出为 CSV / JSON
- CSV 格式：`Name, Timestamp, FrameCount, Value`
- JSON 格式：复用 ExportEntry 结构的扩展版本

### 6.4 与窗口克隆集成

- 克隆窗口时，监视面板的状态（选中项、折叠状态、图表范围）一并复制
- 但监视数据源是全局共享的（WatchManager 单例），不复制数据

### 6.5 与日志文件持久化集成

- 监视历史**可选**写入独立文件（`EnhancedConsole_Watch.log`）
- 复用 `EnhancedConsoleLogFile` 的缓冲写入机制（50 条或 1.5 秒刷新）
- 默认关闭，需在设置中手动开启

## 7. 性能设计

### 7.1 零分配目标

| 场景 | 策略 |
|------|------|
| `Watch.Set(string, float)` | 基本类型重载，无装箱 |
| `Watch.Set(string, object)` | 无法避免装箱，但编辑器中可接受 |
| 字符串格式化 | 缓存上一次 FormattedValue，值未变则跳过格式化 |
| 历史记录 | 预分配环形缓冲区，无 GC |
| UI 刷新 | 复用 ListView BindItem 模式，复用现有 MinRepaintIntervalMs = 50ms 节流 |

### 7.2 条件编译

```csharp
// Watch 类所有方法标记 [Conditional("UNITY_EDITOR")]
// 发布构建中整个调用链被编译器移除
// 无需 #if UNITY_EDITOR 包裹调用代码
```

### 7.3 自动监视更新频率

- 默认每帧采集一次（通过 `EditorApplication.update`）
- 可配置采集间隔（如每 N 帧或每 M 秒采集一次）
- PlayMode 退出时自动暂停所有 Auto 监视

### 7.4 大量监视条目

- ListView 虚拟化（已有基础设施）
- 监视条目上限 1000（可配置）
- 超出上限时拒绝新注册并在 Console 输出警告

## 11. 风险与决策点

| 风险 / 决策 | 分析 | 建议 |
|------------|------|------|
| **Lambda 捕获导致内存泄漏** | `Watch.Auto()` 的 Lambda 可能捕获 MonoBehaviour 引用 | 使用 WeakReference + owner 参数，Auto 监视在 owner 被销毁后自动移除 |
| **主线程限制** | `Watch.Set()` 可能从非主线程调用 | 与现有 `PendingEntries` 机制一致，非主线程写入加锁队列，主线程 flush |
| **条件编译的 Lambda 参数** | `[Conditional]` 是否移除 Lambda 的创建？ | 是。`[Conditional]` 移除整个调用语句，包括参数表达式的求值。已验证。 |
| **与 Inspector Watch 混淆** | Unity 自带 Inspector 可以查看变量 | 定位不同：Inspector 是对象级别的静态查看；我们是代码级别的动态监视 + 历史 |
| **UI Toolkit 图表性能** | 大量数据点的自绘可能卡顿 | 降采样：图表宽度 px 为上限，多点合并（取 min/max） |
| **面板集成方式** | Tab 切换 vs 独立窗口 vs 分割面板 | 建议 Tab 切换（最轻量，不改变窗口布局） |

---

## 12. 非目标（明确排除）

- **代码生成**：不使用 LiveWatch 的 Source Generator 方案，过于复杂
- **远程监视**：不支持从构建中远程发送监视数据（可作为未来扩展）
- **反射自动发现**：不通过反射自动发现 MonoBehaviour 的字段（性能不可控）
- **Serilog / 结构化日志集成**：不属于此功能范围
- **断点 / 条件暂停**：这是调试器的职责，不在此实现
